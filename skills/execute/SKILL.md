# Execute — 태스크 실행 (Wave 병렬 + Acceptance Criteria)

계획에서 다음 실행 가능한 태스크를 찾아 실행한다. 핵심 원칙: **깨끗한 컨텍스트에서, 명확한 계획으로, 적절한 모델로.**

## 사전 조건

- `_workspace/project-plan.json` (status: approved)
- `_workspace/execution-state.json`
- 없으면 안내: "먼저 `/plan:approve`로 계획을 승인해주세요"

## 실행 흐름

### 1. 다음 태스크 선택

execution-state에서 다음 실행할 태스크를 찾는다:
- 의존성이 모두 완료된 태스크 중 가장 앞 순서
- 이미 완료/실패/건너뛴 태스크는 제외

**Wave 병렬 감지** (NEW):
같은 wave 번호를 가진 태스크가 여러 개이고, 모두 의존성이 충족되면:
- "Wave {n}의 태스크 {개수}개를 병렬로 실행할 수 있습니다. 병렬 실행할까요?"
- 사용자 승인 시 Agent 도구로 병렬 실행
- 거부 시 순차 실행

### 2. 컨텍스트 조립

태스크 실행에 필요한 최소한의 컨텍스트만 준비:

```
주입하는 것:
  1. 태스크 계획 (plan 필드) — 무엇을 어떻게 할지
  2. 의존 태스크의 handoff — 이전 작업에서 알아야 할 것만
  3. 프로젝트 파일 트리 — 현재 코드 구조 파악용
  4. 검증 조건 — 완료 후 뭘 확인해야 하는지
  5. 도메인 스킬 — skills 필드에 지정된 전문지식 (있을 때만)
  6. 의사결정 기록 — contexts/sprint-{id}.md (있을 때만)
  7. Acceptance criteria — 완료 기준 (NEW)

주입하지 않는 것:
  - 전체 대화 히스토리
  - 다른 태스크의 상세 내용
  - 이전 세션 로그
```

이렇게 하는 이유: 컨텍스트가 길어지면 AI가 초반 지시를 무시하고 품질이 떨어진다. 매번 깨끗한 상태에서 시작해야 일관된 품질을 유지할 수 있다.

### 3. 모델 배치

태스크의 `model` 필드에 따라 적절한 모델을 사용:
- `haiku`: 단순 작업. 빠르고 저렴.
- `sonnet`: 일반 구현. 균형잡힌 선택.
- `opus`: 복잡한 설계/로직. 최고 품질.

### 4. 실행

#### single 모드 (기본)
해당 모델로 태스크 계획을 직접 실행한다.
코드를 작성하고, 파일을 생성/수정한다.

#### single + skills 모드
single과 동일하되, `skills` 필드에 지정된 도메인 스킬을 컨텍스트에 주입한다.
스킬은 `harnesses/`에서 로드. 검색 경로:
1. `harnesses/*/skills/{skill-name}/skill.md` — 카테고리 전체에서 스킬명 검색
2. 하네스 내부의 에이전트 스킬도 포함
하네스를 통째로 안 써도, 스킬의 전문지식만으로 품질이 올라간다.

#### harness 모드
태스크에 지정된 하네스를 로드하여 멀티에이전트 협업으로 실행한다.
하네스 로드 경로: `harnesses/{category}/{harness-name}/`
하네스 내부의 에이전트들이 역할을 나눠 작업한다.

### 5. Acceptance Criteria 검증 (NEW)

실행이 끝나면, 코드를 작성한 후 다음 단계로 넘어가기 전에 acceptance criteria를 확인:

```
Acceptance Criteria 검증:
  ✅ src/index.html 파일이 존재한다
  ✅ <!DOCTYPE html> 태그가 포함되어 있다
  ❌ <canvas id="game"> 태그가 포함되어 있다
```

하나라도 실패하면:
- 에이전트가 자동으로 수정 시도 (1회)
- 재시도 후에도 실패하면 사용자에게 보고

### 6. 완료 처리

실행이 끝나면:

1. **Handoff 생성**: `/handoff` 스킬을 자동으로 호출하여 작업 이력을 남긴다
2. **자동 검증**: `validation.auto`에 지정된 항목을 실행 (build, typecheck 등)
3. **Acceptance Criteria 확인**: 완료 기준 전체 충족 확인
4. **상태 업데이트**: execution-state.json 갱신
5. **결과 보고**: 사용자에게 간략히 보고

```
✅ 태스크 완료: "HTML 뼈대 만들기" (haiku, 45초, $0.002)
   변경 파일: src/index.html
   검증: build ✅ typecheck ✅
   Acceptance: 3/3 ✅
   
   다음 태스크: "게임 루프 구현" [medium/sonnet]
   → `/execute`로 계속 진행하거나, `/status`로 전체 현황을 확인하세요
```

### 7. 실패 시

검증 실패나 에러 발생 시:
- 실패 원인을 기록
- retry_count 증가
- 사용자에게 알림: "[재시도: `/retry`] [건너뛰기: `/skip`] [상태 확인: `/status`]"

## 스프린트/마일스톤 완료 감지

태스크 완료 후, 해당 스프린트의 모든 태스크가 완료되었으면:
- "스프린트 [이름] 완료! `/validate`로 통합 검증을 실행하세요"라고 안내

마일스톤의 모든 스프린트가 완료되었으면:
- "마일스톤 [이름] 완료! `/audit`로 마일스톤 검증을 실행하세요"라고 안내
