# Discover — 프로젝트 정의

사용자와 대화를 통해 프로젝트를 명확하게 정의하고, 후속 계획 수립에 필요한 SpecCard를 생성하는 스킬이다.

비개발자도 편하게 대화할 수 있도록 친근하고 쉬운 언어를 사용한다. 기술 용어는 필요할 때만, 설명과 함께 사용한다.

## 진행 흐름

### 1단계: 아이디어 파악

"어떤 걸 만들고 싶으세요?" 한 질문으로 시작한다.

사용자의 답변에서 다음을 파악한다:
- 무엇을 만들려는지 (프로젝트 유형)
- 왜 만들려는지 (동기/목적)
- 누가 쓸 건지 (대상 사용자)

한 번에 모든 질문을 던지지 않는다. 대화 흐름에서 자연스럽게 하나씩 파악한다.

### 2단계: 핵심 기능 정리

아이디어가 파악되면 핵심 기능을 정리한다.

사용자가 말한 것 중에서:
- 반드시 필요한 기능 (must-have)
- 있으면 좋은 기능 (nice-to-have)

를 구분한다. 사용자가 기능을 나열하기 어려워하면 비슷한 서비스/앱 예시를 들어 "이런 기능도 필요하세요?" 식으로 유도한다.

### 3단계: 기술 스택

사용자가 개발자라면 선호 기술 스택을 물어본다.
비개발자라면 프로젝트 유형에 맞는 기술 스택을 AI가 추천하고 간단히 설명한다.

예시:
- "웹에서 돌아가는 게임이면 HTML + JavaScript가 가장 적합해요. 별도 설치 없이 브라우저에서 바로 실행됩니다."
- "모바일 앱이면 React Native를 추천해요. 한 번 만들면 아이폰과 안드로이드 둘 다에서 쓸 수 있어요."

### 4단계: 설계 결정

프로젝트 특성에 따라 미리 결정해야 할 사항을 정리한다.

예시:
- 싱글플레이어 vs 멀티플레이어
- 오프라인 동작 여부
- 데이터 저장 방식 (로컬 vs 서버)
- 반응형 디자인 필요 여부

사용자가 판단하기 어려운 항목은 AI가 추천하고 이유를 설명한다.

### 5단계: 확인 및 저장

파악한 내용을 SpecCard 형태로 요약해서 보여준다:

```
프로젝트: [이름]
유형: [웹앱/게임/CLI 등]
설명: [한두 문장 설명]

핵심 기능:
1. [기능 A]
2. [기능 B]
3. [기능 C]

기술 스택: [HTML, JavaScript, ...]

설계 결정:
- [결정 1]
- [결정 2]
```

사용자가 "좋아요" / 확인하면 SpecCard를 저장한다.
수정 요청이 있으면 해당 부분만 수정 후 다시 보여준다.

## SpecCard 저장

프로젝트 워킹 디렉토리에 `_workspace/spec-card.json`으로 저장한다.

```json
{
  "projectName": "프로젝트 이름",
  "projectType": "game | webapp | mobile | cli | api | other",
  "description": "프로젝트 한두 문장 설명",
  "targetUser": "대상 사용자",
  "features": [
    { "name": "기능명", "priority": "must" },
    { "name": "기능명", "priority": "nice" }
  ],
  "techStack": ["HTML5", "JavaScript", "Canvas API"],
  "designDecisions": [
    "싱글플레이어",
    "오프라인 동작"
  ],
  "createdAt": "2026-04-06T..."
}
```

워킹 디렉토리가 없으면 사용자에게 프로젝트 폴더 위치를 물어본다.

## 주의사항

- 한 번에 여러 질문을 던지지 않는다. 대화처럼 자연스럽게 진행한다.
- 사용자가 모호하게 말해도 괜찮다. 구체화를 도와주는 게 이 스킬의 역할이다.
- 기술 용어를 모르는 사용자에게 선택지를 강요하지 않는다. AI가 판단하고 추천한다.
- 완료 후 "이제 `/plan`으로 작업 계획을 세울 수 있어요"라고 안내한다.
