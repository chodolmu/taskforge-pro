# Discuss — 회색지대 식별 + 의사결정 기록

스프린트 실행 전, 구현에서 여러 방향으로 갈 수 있는 모호한 부분(회색지대)을 식별하고 사용자와 논의하여 결정을 기록한다.

**사용자 = 비전을 가진 기획자. AI = 구현하는 빌더.**
사용자에게 기술적 질문을 하지 않는다. 비전과 선택지에 대해서만 묻는다.

## 사전 조건

- `_workspace/project-plan.json` (status: approved)
- `_workspace/execution-state.json`
- 없으면 안내: "먼저 `/plan:approve`로 계획을 승인해주세요"

## 진행 흐름

### 1. 대상 스프린트 확인

인자가 있으면 해당 스프린트, 없으면 다음 실행할 스프린트를 대상으로 한다.
기존 context 파일이 있으면 "이전 논의를 이어서 할까요, 새로 시작할까요?"를 물어본다.

### 2. 기존 결정 로드

이전 스프린트의 context 파일들을 읽어 이미 결정된 사항을 파악한다.
이미 결정된 것을 다시 묻지 않는다.

### 3. 회색지대 분석

스프린트의 태스크들을 분석하여 회색지대를 도출한다.

**회색지대란**: 여러 방향으로 구현 가능하고, 선택에 따라 결과가 달라지는 부분.

회색지대 예시 (구체적이어야 함):
- "카드 레이아웃 vs 리스트 레이아웃" → 카드 레이아웃 결정
- "무한 스크롤 vs 페이지네이션" → 모바일은 무한 스크롤, 데스크톱은 페이지네이션
- "에러 시 재시도 vs 즉시 실패" → 네트워크 에러만 3회 재시도

**회색지대가 아닌 것** (묻지 않음):
- 코드 패턴 선택 (AI가 결정)
- 기술적 위험 (리서치가 판단)
- 성공 지표 (작업에서 자연스럽게 도출)

### 4. 사용자에게 제시

```
## 스프린트 [이름] — 논의할 사항

다음 중 논의하고 싶은 항목을 선택해주세요 (번호 또는 "전부"):

1. 🔲 카드 레이아웃 vs 리스트 레이아웃
2. 🔲 빈 상태(empty state) 표시 방식
3. 🔲 로딩 중 표시 (스켈레톤 vs 스피너)
4. 🔲 AI 재량: 버튼 색상, 여백 등 세부 스타일

선택하지 않은 항목은 AI 재량으로 결정합니다.
```

### 5. 심화 논의

선택된 항목마다 깊이 있게 대화한다.
한 번에 하나씩 진행하며, "결정됐다"고 느끼면 다음으로 넘어간다.

### 6. 의사결정 기록 저장

`_workspace/contexts/sprint-{id}.md`에 저장:

```markdown
---
sprint: m1-s1
status: complete
created: 2026-04-07T...
---

# Sprint m1-s1 — 의사결정 기록

## 확정된 결정 (Locked)

### 레이아웃
- **카드 레이아웃** 사용 (2열, 모바일은 1열)

### 빈 상태
- 일러스트 + "아직 항목이 없습니다" 메시지

## AI 재량 (Discretion)

- 버튼 색상, 여백, 폰트 크기 등 세부 스타일
- 로딩 스켈레톤 구체적 형태
```

## 범위 가드레일

**허용**: 스프린트 범위 안에서 "어떻게" 구현할지 명확화
**금지**: 새 기능 추가 (범위 확장)

사용자가 새 기능을 제안하면:
> "[기능 X]는 새로운 기능이네요 — 별도 스프린트로 다루는 게 좋겠습니다. 지금은 현재 스프린트에 집중할까요?"

## 완료 후

"의사결정이 기록되었습니다. `/execute`로 실행을 시작하세요"라고 안내한다.
