# Verify — 대화형 UAT

스프린트 완료 후 사용자와 대화하며 기능을 하나씩 확인한다.
AI가 "이렇게 동작해야 합니다"를 안내하고, 사용자가 확인하여 결과를 기록한다.

## 철학

**기대 동작을 보여주고, 현실과 맞는지 물어본다.**

AI가 제시: "여기서 버튼을 클릭하면 모달이 열려야 합니다."
사용자 응답:
- "네" / "맞아" / "다음" → pass
- 그 외 → issue로 기록, 심각도는 AI가 자동 추론 (절대 물어보지 않음)

## 사전 조건

- 대상 스프린트의 태스크가 모두 완료 상태
- 없으면 안내: "먼저 스프린트를 완료해주세요"

## 진행 흐름

### 1. 테스트 항목 추출

스프린트의 handoff 파일들에서 테스트 가능한 항목을 추출한다.
서버/DB 변경이 포함된 경우, 콜드 스타트 스모크 테스트를 맨 앞에 추가한다.

### 2. 기존 세션 확인

`_workspace/validations/uat-sprint-{id}.md`가 이미 있으면:
- "이전 UAT를 이어서 할까요, 새로 시작할까요?"

### 3. 테스트 진행

한 번에 하나씩 테스트를 진행한다:

```
## 테스트 1/6: 메인 페이지 로딩

**기대 동작**: 브라우저에서 localhost:3000을 열면 카드 레이아웃의 메인 페이지가 표시됩니다.
- 상단에 네비게이션 바
- 2열 카드 그리드
- 각 카드에 제목과 설명

확인해주세요 →
```

### 4. 결과 기록

사용자 응답에 따라:

- **pass**: 다음 테스트로 이동
- **issue**: 문제 내용 기록 + 심각도 자동 추론

심각도 기준 (자동):
- **blocker**: 핵심 기능 불가 (페이지 안 열림, 크래시)
- **major**: 주요 기능 오동작 (데이터 안 보임, 버튼 미작동)
- **minor**: 부차 기능 이상 (정렬 깨짐, 색상 다름)
- **cosmetic**: 미관 문제 (여백, 폰트 크기)

### 5. 진단 (이슈 발견 시)

이슈가 있으면 병렬 디버그 에이전트를 실행하여 원인을 조사한다.
조사 결과를 UAT 파일에 기록한다.

### 6. UAT 파일 저장

`_workspace/validations/uat-sprint-{id}.md`에 저장:

```markdown
---
status: complete | partial | testing
sprint: m1-s1
started: 2026-04-07T...
updated: 2026-04-07T...
---

# UAT — Sprint m1-s1

## 결과 요약
- 통과: 4/6
- 이슈: 2건 (major 1, minor 1)

## 테스트 결과

### 1. 메인 페이지 로딩
기대: 카드 레이아웃 메인 페이지 표시
결과: pass

### 2. 카드 클릭
기대: 상세 페이지로 이동
결과: issue (major)
설명: 클릭 시 404 에러 발생
원인: 라우팅 설정에서 상세 페이지 경로 누락
파일: src/routes.js:15

## 미해결 이슈 (Gaps)
- truth: "카드 클릭 시 상세 페이지 이동"
  severity: major
  root_cause: "라우팅 경로 누락"
  artifacts: ["src/routes.js"]
```

### 7. 후속 조치

- 이슈 없음: "UAT 통과! 다음 스프린트로 진행하거나 `/audit`으로 마일스톤 검증을 할 수 있습니다"
- 이슈 있음: "이슈 N건이 발견되었습니다. `/retry`로 수정하거나, `/plan:edit`로 수정 태스크를 추가할 수 있습니다"
- blocker 있음: "블로커 이슈가 있어 다음 단계로 진행하기 전에 해결이 필요합니다"
