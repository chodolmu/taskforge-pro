# Refresh — 계획 갱신 (Opus PM)

스프린트 완료 후, 실제 작업 결과를 반영하여 후속 태스크의 계획을 갱신한다.
Opus 모델로 실행한다.

## 왜 필요한가

계획은 실행 전에 세운 것이다. 실제 구현하면서:
- 설계가 바뀔 수 있다 ("REST API 대신 WebSocket 사용")
- 예상 못한 문제가 발견될 수 있다
- 태스크가 추가/삭제될 수 있다

이런 변경사항을 후속 태스크에 반영하지 않으면, 후속 태스크가 잘못된 전제로 작업하게 된다.

## 동작

### 입력
1. 완료된 스프린트의 모든 handoff (설계 결정, 이슈 등)
2. 남은 태스크/스프린트 목록
3. 원래 계획 (project-plan.json)

### Opus가 수행하는 것
1. handoff에서 설계 변경사항 추출
2. 후속 태스크의 `plan` 필드에 변경사항 반영
3. 필요시:
   - 태스크 추가 (새로 발견된 작업)
   - 태스크 삭제 (불필요해진 작업)
   - 태스크 수정 (계획 변경)
   - 의존성 갱신
4. 변경 사유 기록

### 출력

갱신된 `_workspace/project-plan.json` + 변경 요약:

```
계획 갱신 완료 (스프린트 1.1 기반)

변경사항:
  수정: m1-s2-t1 "공룡 렌더링" — Canvas context 전역 캐시 반영
  추가: m1-s2-t4 "더블 점프 방지" — 테스트 중 발견된 이슈
  삭제: (없음)

영향받는 태스크: 3개
→ `/status`로 갱신된 계획을 확인하세요
```

## 자동 트리거

`/validate sprint` 통과 후 자동으로 "계획 갱신이 필요할 수 있습니다. `/refresh`를 실행하시겠어요?"라고 안내한다.
