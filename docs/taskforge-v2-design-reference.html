1. Vision — 이 도구가 해결하려는 것

TaskForge v2는 사람 PM을 흉내내는 도구가 아니다. AI의 망각과 세션 단절을 파일시스템으로 보상하는 도구다.

비개발자도 Claude Code로 큰 프로젝트 (특히 게임) 을 완성할 수 있게 하는 것. 핵심은:

  • AI는 매 세션 기억이 증발한다 → 파일이 기억을 대신한다
  • AI는 빈칸 추측을 못한다 → spec + 레퍼런스가 의도를 고정한다
  • AI는 35분 지나면 품질이 떨어진다 → 작업은 작게, 검증은 자주
  • 사용자는 개발자가 아닐 수 있다 → 용어는 숨기고 질문으로만 수집

2. 문제 인식 — 기존 TaskForge의 한계

문제 1: 큰 프로젝트 진행 불가

Discover 한 번으로 마일스톤 5개, 태스크 80개를 앞에서 다 정의하려 함. 현실에선 그 시점엔 결정할 수 없는 것이 너무 많음. 결과: Opus PM이 3번째 마일스톤부터 헛소리 시작.

문제 2: 작은 작업에도 오버엔지니어링

CSS 한 줄 수정에도 discover→plan→execute 풀 플로우를 돌림. "이 도구 쓰느니 내가 직접 하는 게 빠른데?" 순간 발생.

근거: Martin Fowler / Marmelab 실측 — Kiro에서 간단한 날짜 표시 기능에 8 파일 1300줄 spec 생성. "sledgehammer to crack a nut" 문제는 Kiro/Spec Kit/BMAD 전부의 약점.

문제 3: 자연어 spec만으론 의도 전달 부족

"카드 게임 UI 만들어줘"로는 AI가 Slay the Spire인지 Hearthstone인지 모름. 자연어는 노이즈가 낌.

근거: CodePromptEval 벤치마크 — GPT-4o가 few-shot 추가로 47.1% → 57.5% pass@1 (+10.4%p). 코드 생성에서 특히 효과적 — naming patterns, error handling, abstractions, test structure 같은 컨벤션을 암묵적으로 전달.

문제 4: 세션 간 망각

태스크 2가 태스크 1의 결정을 모름. Handoff가 "의존성 있을 때만"이라 누락됨.

근거: "Why Your Multi-Agent Chain Fails at Agent 3" — 3번째 에이전트부터 컨텍스트 유실. 명시적 handoff 없이 "앞선 에이전트가 알려줬겠지" 가정이 깨짐.

문제 5: PM이 "Opus니까 똑똑하다" 가정

강제 체크 규칙이 없어서 PM도 빈틈 많은 spec을 통과시킴.

3. 핵심 원칙 — AI 친화적이라는 것

사람 파이프라인과 AI 파이프라인의 차이:

영역사람 방식AI 친화적 방식
계획앞에 다 짜고 시작JIT — 필요할 때만 상세화
문서참고용정본(source of truth)
회의대화로 동기화명시적 handoff 파일
결정암묵적즉시 decisions/ 기록
집중8시간 지속35분 태스크 상한
컨텍스트누적됨매 세션 리셋, 파일로 복원
검증팀 리뷰self-verifier 체크리스트

8개 설계 원칙

  1. JIT 계획 — Vision만 고정, 나머지는 필요한 시점에 상세화
  2. 파일시스템 = 기억 — 모든 맥락을 파일로, 세션 간 증발 방지
  3. Spec은 계약 — outcomes / scope / constraints / prior decisions / tasks / verification 6요소 필수
  4. 크기별 루트 분기 — tiny / normal / feature / milestone 4단계, 오버엔지니어링 방지
  5. 명시적 Handoff — 조건부가 아닌 기본값. tiny만 스킵 허용
  6. 자연어보다 레퍼런스 — 작동하는 예시가 10줄 spec보다 강하다
  7. 복잡한 결정엔 CoT 스캐폴드 — "알아서 해" 금지, 생각 순서 제공
  8. 중요 산출물은 Verifier로 재검증 — 자연어 체크리스트로 self-verify

4. 새 워크플로우 — 3층 구조 + JIT

전체 플로우

Phase 0. VISION         → vision.json (고정, 안 바뀜)
   ↓
Phase 1. CONCEPT        → concept.json (교체 가능, M0 retro에서만 수정)
   ↓
Phase 2. ROADMAP        → roadmap.json (느슨한 sketch, 마일스톤 리스트만)
   ↓
Phase 3. M0 PROTOTYPE   → 재미/가능성 검증 + 레퍼런스 생산
   ↓  ← M0 retro에서 concept 뒤집기 가능
Phase 4. MILESTONE LOOP (반복)
   ├─ discover {M}    (그 마일스톤만 sketch → full로 승격)
   ├─ plan {M}        (스프린트/태스크 분해)
   ├─ execute         (태스크 실행)
   ├─ validate        (자동 검증)
   ├─ playtest/UAT    (사용자가 직접 확인, 정성 피드백)
   └─ retro           (배운 것 → roadmap 갱신)
   ↓
Phase 5. POLISH & SHIP

3층 구조 — 핵심 아이디어

층파일성격변경 가능성
Visionvision.json프로젝트의 "왜"안 바뀜 (바뀌면 새 프로젝트)
Conceptconcept.json구현 방향 (장르, 코어 루프)M0 retro에서만 가능
Roadmaproadmap.json마일스톤 리스트매 retro마다 갱신
Planproject-plan.json현재 마일스톤의 태스크현재 active 한 개만 full

마일스톤 상세 레벨 3단계

  • name-only — 이름만. "언젠가 할 것."
  • sketch — 목적, 대략 범위, openQuestions. plan은 없음.
  • full — spec-card + plan 완전 분해. 현재 active 마일스톤만 이 상태.

5. 7개 개선안 — 무엇을 추가할 것인가

① Vision / Concept / Roadmap 3층 분리 + JIT Discover P0 사용자 가시

현재 마일스톤만 full spec. 나머지는 sketch. 각 마일스톤 끝에서 retro → 다음 마일스톤 discover.

근거: HTN (Hierarchical Task Network) — AI 계획의 표준. JIT 루프 "Plan→Act→Reflect→Repeat"는 agentic 계획의 핵심.
반대 근거도 있음: "AI agents benefit from MORE detailed upfront planning than human teams" — AI는 빈칸 추측을 못함.
결론: Vision은 detailed upfront, 마일스톤은 JIT. 균형.

② 작업 크기 분기 게이트 P0 사용자 가시

진입 시 먼저 분류: tiny / normal / feature / milestone.

크기예시루트
tiny (5분)CSS 색 변경/taskforge-quick, spec 없이
normal (35분)버그 수정, 단일 기능acceptance criteria만, spec 없음
feature (여러 태스크)로그인 플로우 전체full SDD (requirements → design → tasks)
milestone (마일스톤 단위)결제 통합retro + replan 루프 포함
근거: Kiro/BMAD 전부의 약점 — "sledgehammer to crack a nut". TaskForge의 차별화 포인트.
단 복잡한 ML classifier는 오버킬. 간단한 규칙 기반으로 충분.

③ References 레이어 (few-shot / CoT / verifier) P0 사용자 가시

3개 디렉토리:

  • references/ — 검증된 프로토타입/샘플 (HTML, 스크린샷, 예시 JSON)
  • prompts/cot/ — CoT 스캐폴드 템플릿 (결정 시 생각 순서)
  • verifiers/ — 자연어 체크리스트

수집은 질문으로:

Q1. 비슷한 게임/앱 있어요? (유튜브 링크, 스크린샷, 게임 이름 OK)
    → references/ 저장
Q2. 디자인 톤 비슷한 이미지 있어요?
    → references/visual/ 저장
Q3. 이것만은 꼭 지켜야 한다는 규칙?
    → constraints.md 저장
Q4. "완성됐다"를 뭘로 판단해요? (3가지)
    → verification.md 저장
근거: CodePromptEval — few-shot이 +10.4%p 성능 향상. 코드에서 특히 강력. AWS Agentic Arcade 사례도 evaluation agent를 별도 station으로 운영.

④ contextManifest (AGENTS.md 표준 준수) P1 내부

태스크마다 "이 파일을 이 순서로 읽어라" 명시. 우리 식 새 포맷 만들지 않고 AGENTS.md 표준 준수 (호환성).

{
  "taskId": "T-M1-003",
  "contextManifest": [
    { "path": "vision.json", "priority": 0 },
    { "path": "concept.json", "priority": 1 },
    { "path": "references/ui-pattern-login.html", "priority": 2 },
    { "path": "decisions/D001-auth-strategy.md", "priority": 3 },
    { "path": "handoffs/T-M1-002.json", "priority": 4 }
  ]
}
근거: AGENTS.md는 이미 Codex/Cursor 등이 채택한 업계 표준. arxiv 2602.20478 "Codified Context: Infrastructure for AI Agents in a Complex Codebase".

⑤ PM 강제 체크리스트 (6-area) P1 내부

PM이 plan 생성 후 자동 검증:

  1. Commands — 실행 명령이 명시됐나?
  2. Testing — 검증 방법이 있나?
  3. Project structure — 파일 위치가 정해졌나?
  4. Code style — 스타일 가이드 참조 있나?
  5. Git workflow — 브랜치/커밋 규칙 있나?
  6. Boundaries — 하면 안 되는 것이 명시됐나?

+ 추가: 태스크 35분 이하인가? verification criteria가 자연어로 체크 가능한가?

근거: Addy Osmani / O'Reilly — "Use the six-area checklist as a sanity check before handing off to the agent". 업계 정설.

⑥ Handoff 기본 ON P1 내부

조건부가 아닌 기본값. tiny 태스크만 스킵 허용.

{
  "taskId": "T-M1-002",
  "completedAt": "2026-04-20T...",
  "whatChanged": ["auth/login.ts 생성", "auth/types.ts 수정"],
  "decisionsMade": ["D001: JWT 대신 세션 쿠키 사용"],
  "hintsForNext": "로그인 성공 시 /dashboard 리다이렉트 필요",
  "openItems": ["비밀번호 재설정 플로우 미구현"]
}
근거: OpenAI Agents SDK / LangChain 모두 handoff를 1급 시민으로 다룸. "Why Your Multi-Agent Chain Fails at Agent 3" — 명시적 handoff 없으면 3번째부터 깨짐.

⑦ 가드레일 (비용/시간 상한) P1 내부

태스크당:

  • maxTurns — 기본 20
  • maxCostUSD — 기본 $2
  • maxWallTimeMin — 기본 35

초과 시 자동 중단 + 현재 상태 저장 + 사용자 확인.

근거: "a single runaway agent can burn through API budgets in hours" — 실제 발생하는 문제. 비용 거의 0, 이득 명확.

6. 버린 아이디어 — 무엇을 추가하지 않을 것인가

기능 경쟁으로 가면 BMAD/Spec Kit의 열화판이 된다. 아래는 명시적으로 거부한 항목들.
아이디어거부 이유
patterns/ 전역 경험 축적 장기적 가치 있지만 비개발자 UX에 안 맞고 정체성 벗어남. 실전 1~2 프로젝트 후 재평가.
별도 Verifier 에이전트 self-verifier 체크리스트로 충분. 복잡도 대비 이득 적음.
Agent Teams (peer-to-peer 통신) Claude Code 자체 기능. 지금 TaskForge엔 오버엔지니어링. 필요하면 위에 얹기.
Living Spec (spec-as-source, Tessl 방식) 야심 찬 방향이지만 과거 MDD 실패 반복 위험. 지금은 하지 않음.
다중 모델 교차검증 비용 2배, 품질 개선 한계.
의존성 자동 탐지 / 영향 추적 IDE 영역. TaskForge가 할 일 아님.
decisions/ 정식화 (1차 릴리스) "중요한 결정만"의 기준이 모호. 쓰다 보면 자연 추가. 1차에선 뺌.

8. 비개발자 UX — 용어 숨기기

원칙: few-shot, CoT, verifier, spec 같은 용어는 절대 노출 안 함. 사용자는 질문에 답하고 파일을 모으기만.

번역표

전문 용어사용자에게 보이는 것
Few-shot prompting"비슷한 예시 모으기" — "이런 느낌으로 만들고 싶어요"
Chain of Thought"생각 순서 정하기" — "이 결정 내릴 땐 뭘 먼저 봐요?"
Verifier"완성 체크리스트" — "이걸 다 통과하면 OK"
Reference"본보기" — "이거랑 비슷하게"
Spec"설계 메모" — 혹은 아예 언급 안 함
Handoff(노출 안 함, 시스템 내부)

CoT 노출 예시 (복잡한 결정 태스크)

이건 중요한 결정이라 [생각 순서]를 먼저 정할게요:
  1. 현재 플레이어 능력치 확인
  2. 비슷한 위치 보스들 비교
  3. 극단 케이스 점검
  4. 최종 제안

이 순서로 진행할까요? [Y/n/순서 수정]

"CoT"라는 말 없이 동일 효과. 다음에 비슷한 태스크 오면 재사용.

Verifier 노출 예시

완성 체크리스트 (verification.md):
□ 로그인 버튼을 눌렀을 때 3초 안에 반응한다
□ 잘못된 비밀번호 입력 시 "다시 확인해주세요" 문구가 뜬다
□ 모바일 화면에서도 버튼이 눌린다

실행 에이전트가 완료 후 이 체크리스트를 스스로 읽고 통과/실패 표시. self-verifier.

9. 리서치 근거 요약

정량 근거

  • Few-shot 효과: 정확도 +10%, F1 +7% (일반), 코드 생성 GPT-4o +10.4%p pass@1
  • AI 성능 저하 시점: 35분 후 degradation 관찰
  • Tool calling 실패율: 실전 3-15%
  • God Agent 라우팅 오류: 20+ 툴 시 잘못된 툴 선택 30%

핵심 실패 모드 (Columbia DAPLab, 2026)

  1. Context loss — 대형 프로젝트에서 아키텍처 이해 상실
  2. Silent errors — placeholder, TODO, skipped test 은폐
  3. Reasoning loops — 같은 실패 반복, 비용 폭주
  4. Security vulns — API key 노출, 접근 제어 미준수
  5. God Agent — 하나의 에이전트가 모든 것을 하려 함
  6. Greenfield 우수 vs Real 실패 — 스니펫은 이해, 시스템은 못함

Spec-Driven Development는 잘못 쓰면 워터폴 복귀다. 실행 전 spec을 다 쓰는 것 = 워터폴. TaskForge도 이 덫을 피해야 한다 — 실행 → 배움 → spec 갱신 루프가 자연스럽게 돌아야 함.

10. 차별화 3요소

경고: "또 다른 SDD 프레임워크"로 수렴하면 패배. BMAD/Spec Kit/Kiro와 기능 경쟁하지 말 것.

TaskForge v2의 차별화는 세 가지로 좁힌다

  1. 한국어 + 비개발자 UX — 용어 숨김, 질문 기반 수집. 경쟁 도구 전부 영어 + 개발자 타겟.
  2. 작업 크기 자동 분기 — tiny/normal/feature/milestone 4단계. 경쟁 도구 전부 "sledgehammer to crack a nut" 문제.
  3. Claude Code 네이티브 — 스킬/서브에이전트/락 기본 활용. 별도 IDE 아닌 Claude Code 위에서 돌아감.
