사용자 개입 불필요. 모든 측정은 telemetry.jsonl 자동 수집. 사용자는 결과만 확인. AI가 스스로 판단해서 진행하는 자동 검증 체계.

1. 검증 원칙 4가지

  1. 가설 먼저, 지표 나중 — "이 숫자가 이 값 이상이면 성공"을 사전 선언
  2. 자동 수집 우선 — 수동 집계가 필요한 지표는 측정하지 않음
  3. 3-Phase 점진 확대 — 1개 → 3개 → 10개 프로젝트 순서
  4. 실패 시 분기 사전 설계 — Phase별로 "실패 시 무엇을 의심할지" 미리 정함

2. telemetry.jsonl 스키마

모든 측정의 근거. _workspace/projects/{id}/telemetry.jsonl에 자동 append.

{"t":"2026-04-20T10:15:00Z","event":"task_start","taskId":"M1-S1-T3","size":"feature","referencesInjected":["ui-pattern.html"],"cotUsed":true,"manifestFiles":5}
{"t":"2026-04-20T10:33:00Z","event":"task_end","taskId":"M1-S1-T3","outcome":"pass","retryCount":0,"tokens":12450,"costUSD":0.42,"wallMin":18}
{"t":"2026-04-20T10:33:05Z","event":"handoff_created","taskId":"M1-S1-T3"}
{"t":"2026-04-20T10:33:10Z","event":"guardrail_triggered","taskId":"M1-S1-T3","type":"maxTurns","value":22,"limit":20}
{"t":"2026-04-20T10:40:00Z","event":"sprint_validate","sprintId":"M1-S1","passed":true,"stubCount":0}
{"t":"2026-04-20T11:00:00Z","event":"milestone_validate","milestoneId":"M1","passed":true}
{"t":"2026-04-20T11:05:00Z","event":"quick_task","description":"버튼 색 변경","outcome":"pass","costUSD":0.001}

집계 예시 (jq)

# 태스크당 평균 재작업
jq -s 'map(select(.event=="task_end")) | (map(.retryCount) | add) / length' telemetry.jsonl

# 첫 시도 pass율
jq -s 'map(select(.event=="task_end")) | (map(select(.retryCount==0)) | length) / length' telemetry.jsonl

# 레퍼런스 수집률
jq -s 'map(select(.event=="task_start")) | (map(select(.referencesInjected | length > 0)) | length) / length' telemetry.jsonl

3. Phase 1 — Smoke Test

P1
Smoke Test
Week 1  ·  소형 게임 프로젝트 1개 (마일스톤 2개, 태스크 15개 이내)
Week 1

목적: 설계가 기본적으로 돌아가는가? 품질 측정 아님.

Go/No-Go 기준

# 가설 측정 방법 임계값 실패 시
S1 vision/concept/roadmap 3층이 생성되고 사용된다 파일 존재 + telemetry task_start에 manifestFiles > 0 100% 설계 수정
S2 작업 크기 분기가 동작한다 tiny → quick, feature → plan 라우팅 정확도 8/10 케이스 분류 로직 재설계
S3 레퍼런스가 실제 프롬프트에 주입된다 telemetry task_start의 referencesInjected 길이 > 0 참조 있는 태스크 100% contextManifest 재설계
S4 Handoff 파일이 기본 생성된다 handoffs/ 파일 수 / 전체 태스크 수 95%+ handoff 로직 확인
S5 가드레일이 작동한다 maxTurns 인위 초과 시 guardrail_triggered 이벤트 발동 확인 즉시 수정
S6 프로젝트가 완주된다 M1 + M2 전부 milestone_validate passed 완주 원인 분석
Phase 1 완료 조건: S1~S5 전부 Go, S6 1회 이상 완주

4. Phase 2 — A/B 비교

P2
A/B 비교
Week 2~3  ·  게임 프로젝트 3개 (소형 / 중형 / 대형)
Week 2~3

목적: v2가 v1보다 실제로 나은가?

대상 프로젝트

프로젝트크기v1 비교 방법
A. 하이퍼캐주얼 마일스톤 2개 v1으로도 한 번 실행
B. 덱빌딩 프로토타입 마일스톤 3~4개 v1 스폿체크
C. 턴제 전략 일부 마일스톤 5+개 v1이 실패하는 지점 확인

핵심 지표 (자동 수집)

# 지표 v1 기준선 (가설) v2 목표 판정 기준
A1 tiny 작업 소요 시간 10~15분 2분 이하 v2가 v1의 30% 이하
A2 feature 첫 시도 pass율 40~55% 55%+ 절대치 55% 이상
A3 태스크당 평균 재작업 2.5~3.5회 2.0회 이하 절대치 2.0 이하
A4 3번째 태스크가 1번째 결정 아는 비율 30~50% 75%+ 절대치 75% 이상
A5 핵심 레퍼런스 수집률 — 70%+ 절대치 70%+ — 가장 중요
A6 M2 도달률 20~30% 60%+ 절대치 60% 이상
핵심 지표: A5 레퍼런스 수집률. 이게 50% 미만이면 다른 지표 다 좋아도 설계 재검토. 개선폭의 50%가 이 UX에 달려 있음.
판정 규칙: A1~A6 중 4개 이상 목표 달성 → Go
Phase 2 완료 조건: 3개 프로젝트 중 2개 이상 Go

P3
실전 검증

2~3개월

목적: 장기 안정성 + 에지 케이스

추가 지표

지표의미임계값

  • 동시 여러 세션 — 락 시스템 현장 검증
  • 장기 프로젝트 (1개월+) — contextManifest 장기 유지
  • 비개발자 사용자 — 레퍼런스 수집 UX 체감

6. 실패 분기 (사전 설계)

Phase별 실패 시 무엇을 의심할지 미리 정함:

Phase 1 실패 → 코드 버그

  • S1~S4 fail: 기본 기능 미작동 → 수정 후 재시도
  • S5 fail: 가드레일 설계 오류 → 즉시 수정
  • S6 fail: 프로젝트 크기 문제 → 더 작은 프로젝트로

Phase 2 실패 → 설계 가설 오류

  • A5 (레퍼런스) fail: 비개발자 UX 가설 부정 → 질문/수집 방식 전면 재설계
  • A2/A3 fail: few-shot 효과 가설 부정 → 레퍼런스 형식 재검토
  • A4 fail: handoff 가설 부정 → 스키마 재설계
  • A6 fail: JIT 가설 부정 → 근본 재검토 (가장 심각)

Phase 3 실패 → 시장 적합성 문제

  • 포기율 높음: 비개발자 타겟 자체 재검토
  • 반복 사용 낮음: 일회성 도구로 인식됨 → patterns/ 도입 재검토

8. 수치 개선 기대값 (사전 선언)

이 수치는 가설이다. Phase 3까지 진행하며 업데이트됨.

영역기존 → v2확신도
tiny 작업 비용 -85% 높음
tiny 작업 시간 -80% 높음
세션 간 결정 연속성 30~50% → 75~90% 높음
feature 첫 시도 pass율 40~55% → 55~70% 중간
마일스톤 M2 도달률 20~30% → 60~75% 중간
재작업 횟수 -40% 중간
비용 예측성 낮음 → 높음 높음 (가드레일)
이 문서는 검증 도구다. 수치는 가설이고, Phase 3까지 진행하며 업데이트됨.
